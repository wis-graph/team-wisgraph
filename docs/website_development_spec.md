## 1. 문서 목적

이 문서는 [team_portfolio_integrated_content_design.md](/Users/saisiot/code_workshop/team-portfolio-2605/docs/team_portfolio_integrated_content_design.md)를 실제 개발에 바로 사용할 수 있는 수준으로 정리한 구현 명세서다.

## 2. 제품 정의

### 제품 유형

- 반응형 단일 랜딩페이지
- 기업형 팀 포트폴리오 사이트
- 한국어 중심, 영문 섹션명 병기

### 핵심 포지셔닝

이 사이트는 교육팀 소개 페이지가 아니라, 다음 인상을 만들어야 한다.

> 업무 지식을 구조화하고, AI Agent와 자동화 워크플로우로 실제 시스템을 구축하는 전문 실행팀

### 주요 사용자

- 공공기관 실무자 및 관리자
- 기업의 리더, 실무 책임자, 혁신 담당자
- AI 도입 검토 중인 팀
- 교육 이후 실제 정착까지 연결하려는 조직

### 핵심 전환 목표

- 프로젝트 상담 문의
- 교육 프로그램 문의

### 비범위

초기 버전에서 아래 항목은 제외한다.

- 블로그
- 뉴스룸
- 다국어 전환
- 로그인 기능
- 관리자 대시보드
- 상세 프로필 페이지
- 상세 사례 페이지

---

## 3. 성공 기준

사이트 1차 버전은 아래 기준을 만족해야 한다.

- 첫 화면에서 "시스템 구축팀" 인상이 명확하다.
- 로고, 아키텍처, 방법론, 구축 역량이 한 흐름으로 읽힌다.
- 멤버 소개보다 역량과 구조가 먼저 보인다.
- 교육 섹션이 강사 소개나 강의 목록처럼 보이지 않는다.
- CTA가 명확하고 부담 없는 상담 흐름을 만든다.
- 데스크톱과 모바일 모두에서 레이아웃이 무너지지 않는다.

## 5. 섹션별 구현 요구사항

### 5.1 Opening Statement

#### 목적

- 첫 화면에서 팀의 정체성을 즉시 전달한다.
- 교육팀, 강사팀, 노코드 홍보 페이지처럼 보이지 않게 한다.

#### 필수 콘텐츠

- 메인 카피: `업무 지식을 AI가 실행할 수 있는 시스템으로.`
- 서브 카피 2~3줄
- 보조 카피 1줄
- CTA 버튼 2개
- 시스템 아키텍처 다이어그램 1개

#### UI 요구사항

- 화면 진입 시 거의 한 화면을 채우는 히어로 구조
- 텍스트 블록과 다이어그램이 분리된 2열 레이아웃
- 모바일에서는 텍스트 우선, 다이어그램은 하단 배치
- 사람 사진, 멤버 카드, 로고 그리드는 첫 화면에 두지 않음

#### 인터랙션

- 다이어그램 선 연결 또는 노드 점등 정도의 절제된 모션만 허용
- CTA hover는 색상 또는 보더 강조 수준으로 제한

---

### 5.2 Clients & References

#### 목적

- 초반 신뢰 형성
- 과장 없이 경험 범위를 보여주기

#### 필수 콘텐츠

- 섹션 타이틀
- 짧은 설명 문장
- 주요 로고 8개 이상
- 안전 문구 1개

#### UI 요구사항

- 모노톤 로고 그리드
- 4열 또는 6열 레이아웃
- 각 로고는 동일한 박스 기준에 맞춤
- 너무 많은 로고를 첫 화면 높이에 몰아넣지 않음

#### 문구 제약

- `공식 파트너`
- `인증 파트너`
- `구축 완료 고객사`
- `Trusted by` 단독 표기

위 표현은 사용하지 않는다.

안전 문구는 아래 성격을 유지한다.

- 일부 레퍼런스는 팀 멤버 개인의 강의, 컨설팅, 코칭, 프로젝트 참여, 운영 경험을 포함한다.

---

### 5.3 Capabilities

#### 목적

- 팀이 실제로 구축할 수 있는 범위를 명확히 보여준다.

#### 필수 카드

- `01 Business Ontology`
- `02 Agent Architecture`
- `03 Workflow Automation`
- `04 Internal Tools & MVP`
- `05 AI Education & Enablement`

#### UI 요구사항

- 5개의 번호 기반 카드
- 카드마다 영문 타이틀, 한국어 설명, 태그 노출
- 카드 하단에 4~6개 태그 또는 짧은 키워드 표시
- 아이콘보다 숫자와 타이포 중심

#### 레이아웃

- 데스크톱에서는 2열 또는 3열 그리드
- 모바일에서는 1열 스택
- 각 카드 높이는 가능한 한 균일하게 맞춤

---

### 5.4 System Architecture

#### 목적

- 팀이 만드는 결과물이 실제 시스템이라는 점을 시각적으로 증명한다.

#### 필수 레이어

- Input Layer
- Ontology Layer
- Agent Layer
- Workflow Layer
- Operation Layer

#### 필수 콘텐츠

- 섹션 타이틀
- 요약 카피
- 5단 레이어 구조
- 각 레이어 설명

#### UI 요구사항

- 수평 레이어 5개를 계단형 또는 수직 스택형으로 구현
- 각 레이어는 얇은 라인과 작은 노드로 연결
- `Ontology Layer`와 `Agent Layer`를 시각적으로 가장 강조

#### 인터랙션

- 데스크톱에서는 hover 시 레이어 설명 노출
- 모바일에서는 tap 또는 아코디언 방식으로 설명 노출
- 인터랙션 없이도 기본 내용은 읽을 수 있어야 함

---

### 5.5 Method

#### 목적

- 구조화 중심 접근법을 전달한다.

#### 필수 단계

- `01 Observe`
- `02 Decompose`
- `03 Model`
- `04 Architect`
- `05 Build`
- `06 Enable`

#### UI 요구사항

- 6단계 타임라인 또는 프로세스 레일
- 숫자, 동사, 짧은 설명의 3요소 구조
- 세부 bullet은 기본 상태에서 3~5개까지만 노출하고, 길어지면 펼침 처리 가능

#### 인터랙션

- 스크롤 진입 시 단계가 순차적으로 나타나는 정도의 가벼운 모션
- 과한 패럴랙스, 대형 Lottie, 컬러풀한 단계별 변화는 금지

---

### 5.6 Client Work

#### 목적

- 업종별 사례가 아니라 업무 구조별 적용 가능 영역을 보여준다.

#### 필수 Workstream

- Public Sector AX
- Enterprise Automation
- Sales & Marketing Intelligence
- Education & Knowledge Systems
- Field Operations & CX

#### UI 요구사항

- 각 Workstream은 큰 제목, 1문장 설명, 작업 리스트를 가진 카드
- 작업 리스트는 4~6개까지만 노출
- 태그형 또는 짧은 리스트형 표현 허용

#### 레이아웃

- 데스크톱 2열 또는 3열 그리드
- 모바일 1열 스택

---

### 5.7 Education & Enablement

#### 목적

- 교육을 독립 서비스가 아니라 조직 정착 역량으로 보여준다.

#### 필수 콘텐츠

- 메인 카피
- 본문 카피
- 대상별 카드 5개
- 프로그램 예시 3~4개

#### 대상 카드

- AI 초보자
- 일반 실무자
- 팀장·관리자
- 임원·고위직
- 기획·개발 실무자

#### UI 요구사항

- `강의 목록`처럼 보이지 않는 카드형 구조
- 각 카드에 대상, 목표, 예시 산출물 또는 학습 범위 표시
- 프로그램 예시는 별도 리스트 또는 콤팩트 카드로 분리 가능

---

### 5.8 Team of Specialists

#### 목적

- 멤버 이력보다 역할 분업 구조를 보여준다.

#### 필수 멤버

- 권용석
- 강동민
- 문찬원
- 김혜련
- 김모세
- 김정은

#### UI 요구사항

- 멤버당 카드 1개
- 이름, 역할명, 1문장 설명, 키워드 노출
- 사진이 있으면 흑백 또는 어두운 톤 보정
- 사진이 없으면 이니셜 또는 텍스트 기반 카드 사용

#### 제약

- 장문 약력은 메인 페이지에 넣지 않음
- 과도한 개인 브랜딩으로 보이지 않게 함

---

### 5.9 Engagements

#### 목적

- 실제 의뢰 가능한 프로젝트 형태를 제시한다.

#### 필수 Engagement

- AI Opportunity Scan
- Ontology Design Sprint
- Agent Automation Build
- AI Enablement Program

#### 카드 필수 항목

- 이름
- 목적 1문장
- 주요 활동 3~5개
- 산출물 3~5개

#### UI 요구사항

- 가격표처럼 보이지 않는 큰 카드
- 서비스 판매보다 프로젝트 설계 흐름처럼 보이게 구성

---

### 5.10 Contact

#### 목적

- 문의 전환

#### 필수 콘텐츠

- 마무리 카피
- 본문 카피
- CTA 버튼 2개
- 문의 폼

#### 문의 폼 항목

- 이름
- 소속/회사
- 직책
- 연락처
- 관심 분야
- 현재 고민
- 희망 일정

#### 관심 분야 옵션

- AI 교육
- 업무 자동화
- Agent 시스템 구축
- 업무 온톨로지 설계
- MVP/내부 업무도구 개발
- 임원/리더 교육

#### 구현 메모

- 초기 버전에서는 실제 제출 기능 없이 `mailto`, 외부 폼, 또는 임시 endpoint 연결로 시작 가능
- 최종 방식은 별도 결정

## 6. 공통 컴포넌트 명세

### Section Header

- 영문 섹션명
- 한국어 보조 설명
- 짧은 본문 1~3문단

### Primary Button

- 용도: 주요 전환
- 예시: `시스템 구축 문의`, `프로젝트 상담하기`
- 스타일: 강조색 채움, 너무 둥글지 않은 모서리

### Secondary Button

- 용도: 보조 전환
- 예시: `교육 프로그램 문의`, `팀 역량 보기`
- 스타일: 투명 배경, 얇은 보더

### Numbered Card

- 용도: Capability, Method, Engagement 등
- 구조: 큰 숫자, 제목, 짧은 설명, 태그 또는 리스트

### Tag Chip

- 용도: 작업 항목, 키워드, 관심 분야
- 스타일: 채우기보다 얇은 보더 우선

### Layer Diagram Item

- 용도: System Architecture
- 구조: 레이어명, 하위 항목, 설명

### Team Card

- 구조: 이름, 역할명, 설명, 키워드

### Contact Field

- 다크 배경과 구분되는 명확한 보더
- 포커스 상태가 분명해야 함

## 7. 디자인 시스템 구현 기준

### 컬러 토큰

```css
:root {
  --bg: #05070a;
  --bg-secondary: #0b1020;
  --surface: #171b26;
  --text-primary: #e8edf2;
  --text-secondary: #8b95a1;
  --accent-primary: #3b82f6;
  --accent-secondary: #20c997;
  --border: #293241;
}
```

### 타이포그래피

- 영문: `Inter`, `Space Grotesk`, `IBM Plex Sans` 계열 중 선택
- 한글: `Pretendard`, `SUIT`, `Noto Sans KR` 계열 중 선택
- 제목은 짧고 크게
- 본문은 줄간격을 넉넉하게
- 영어 섹션명 + 한국어 설명 조합 유지

### 레이아웃

- 최대 콘텐츠 폭은 넓게 잡되, 본문 가독성 폭은 제한
- 섹션별 상하 여백을 충분히 사용
- 카드 모서리는 과하게 둥글지 않게 유지
- 카드와 섹션 경계는 그림자보다 보더와 대비로 해결

### 그래픽

- 노드 그래프
- 레이어 다이어그램
- 워크플로우 라인
- 얇은 그리드
- 미세한 노이즈 텍스처

### 금지 요소

- 귀여운 캐릭터
- AI 로봇 이미지
- 컬러풀한 SaaS 일러스트
- 말랑한 버튼
- 과한 그라디언트
- 과한 애니메이션
- 클립아트 아이콘

## 8. 반응형 규칙

### 데스크톱

- Hero는 2열
- Capability, Client Work, Team, Engagement는 그리드 중심
- Architecture는 수평 레이어 표현 우선

### 태블릿

- 2열 레이아웃 유지 가능
- 로고 그리드는 4열 이하로 축소
- Architecture 설명은 레이어 하단 또는 옆에 고정 배치 가능

### 모바일

- 모든 섹션은 1열 우선
- Hero 다이어그램은 텍스트 아래
- Method는 수직 타임라인
- Architecture는 아코디언 또는 카드 스택
- Contact 폼은 큰 입력 영역과 충분한 터치 간격 보장

---

## 9. 모션 규칙

- 페이지 로드 시 큰 한 번의 분위기 연출보다, 각 섹션의 가벼운 reveal 중심
- hover는 보더 강조, 색상 변화, 미세한 이동 정도만 허용
- Architecture 선 연결은 느리고 절제된 방식으로 구현
- 사용자의 주의를 빼앗는 반복 애니메이션은 금지

---

## 10. 콘텐츠 운영 규칙

- 한 섹션에 텍스트를 과도하게 넣지 않는다.
- 긴 설명보다 구조와 메시지 우선으로 편집한다.
- 교육 문구는 실무 정착과 조직 적용 맥락을 유지한다.
- 팀 소개는 개인 홍보보다 협업 구조를 우선한다.
- 로고와 레퍼런스는 과장 없이 보수적으로 표현한다.

## 11. 접근성 및 SEO 기준

### 접근성

- 모든 버튼과 링크는 키보드 포커스 가능
- 텍스트 대비는 WCAG AA 수준 이상 목표
- 로고 이미지와 장식 그래픽은 적절히 `alt` 구분
- hover 전용 정보는 모바일에서도 접근 가능해야 함

### SEO

- 단일 명확한 `h1` 사용
- 섹션별 `h2`, 카드 내부는 `h3` 계층 유지
- `title`, `meta description`, Open Graph 기본값 설정
- 주요 키워드는 `AI Agent`, `업무 자동화`, `온톨로지`, `AX`, `시스템 구축` 축으로 정리

## 14. 완료 기준 체크리스트

- 메인 페이지 10개 섹션이 모두 구현되어 있다.
- 첫 화면에 팀 정체성이 명확히 드러난다.
- 로고, 역량, 아키텍처, 방법론 흐름이 끊기지 않는다.
- 디자인이 밝고 가벼운 SaaS 템플릿처럼 보이지 않는다.
- 교육 섹션이 강의 판매 페이지처럼 보이지 않는다.
- 모바일에서 모든 섹션이 읽기 가능하다.
- Contact CTA와 문의 폼 흐름이 동작한다.
- 텍스트, 링크, 버튼 포커스 상태가 확인된다.
